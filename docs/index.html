Local-first kanban for human work and bounded agent updates.

BacBan is a real local board: a React frontend, an Express JSON API, Docker Compose, ignored runtime board data, and a documented write contract for Codex-style assistants.

What the app actually supports today.

The public demo page describes the current board surface rather than inventing a future dashboard. The running app is the source of truth.

Boards and columns

Work and Life boards, configurable column order/titles, collapsed board state, and persistent board ordering.

Cards

Card text, project color, due date, references, `waitingOn`, `updatedAt`, `doneAt`, and recent-change highlighting.

Nested subtasks

Card-level To Do / Doing / Done subtask lists with a mobile-friendly nested view.

Local persistence

The backend writes the whole board document to local JSON. The API intentionally stays small and auditable.

Run a safe demo without touching live board data.

The repository includes a tracked demo seed with Work and Life boards, sample cards, due dates, references, waiting-on status, and subtasks. It is copied into ignored runtime state only when you explicitly seed it.

  • `demo/kanban-data.demo.json` is safe to publish.
  • `kanban-data/` remains ignored and should contain only local runtime data.
  • `docs/DEMO.md` shows how to seed, back up, and reset.
git clone https://github.com/skittixch/bacban.git
cd bacban
docker compose up -d --build
Invoke-RestMethod -Uri 'http://127.0.0.1:3001/health'

Agent writes are deliberately boring.

BacBan does not ask an assistant to improvise persistence. The write path is a simple checklist with evidence and readback.

  1. 01 Back up

    Copy the runtime JSON before changing board state.

  2. 02 Read full state

    GET `/api/data` and match existing cards before creating new ones.

  3. 03 Post full state

    Modify only needed fields, validate JSON, then POST the full document.

  4. 04 Read back

    Verify health and the changed card before reporting completion or blocked state.

BacBan is private-first until local runbooks and operator-specific details are sanitized for public release. Publishing checklist
